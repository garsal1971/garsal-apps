FIX — Cron job fill-notification-queue usa Vault invece di current_setting
Il cron che chiama la edge function fill-notification-queue fallisce con errore unrecognized configuration parameter "app.service_role_key" perché legge il secret tramite current_setting() invece di vault.decrypted_secrets. Questo SQL corregge lo scheduling.
SQL PROD pg_cron · Vault 
-- 1. Rimuove il job precedente (se esiste)
SELECT cron.unschedule('fill-notification-queue');

-- 2. Ricrea il job usando vault.decrypted_secrets per leggere il secret
SELECT cron.schedule(
  'fill-notification-queue',
  '* * * * *',   -- ogni minuto
  $$
  SELECT net.http_post(
    url  := 'https://jajlmmdsjlvzgcxiiypk.supabase.co/functions/v1/fill-notification-queue',
    headers := jsonb_build_object(
      'Content-Type',  'application/json',
      'Authorization', 'Bearer ' || (
        SELECT decrypted_secret
        FROM   vault.decrypted_secrets
        WHERE  name = 'service_role_key'
        LIMIT  1
      )
    ),
    body := '{}'::jsonb
  );
  $$
);

-- 3. Verifica: il job deve apparire con schedule '* * * * *'
SELECT jobid, jobname, schedule, active
FROM   cron.job
WHERE  jobname = 'fill-notification-queue';
FIX
FIX — DecisionKeeper: RLS su dc_decisions e dc_logs
La tabella dc_decisions non filtra per user_id e dc_logs non ha RLS abilitato. Questo SQL aggiunge le policy necessarie per isolare i dati per utente autenticato.
SQL PROD RLS · dc_decisions · dc_logs 
-- ── dc_decisions ──────────────────────────────────────────────────────────
ALTER TABLE dc_decisions ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS "dc_decisions_user" ON dc_decisions;
CREATE POLICY "dc_decisions_user" ON dc_decisions
  FOR ALL
  USING  (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- ── dc_logs (FK → dc_decisions, nessun user_id diretto) ────────────────
ALTER TABLE dc_logs ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS "dc_logs_via_decisions" ON dc_logs;
CREATE POLICY "dc_logs_via_decisions" ON dc_logs
  FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM dc_decisions d
      WHERE d.id = dc_logs.decision_id
        AND d.user_id = auth.uid()
    )
  );

-- Verifica
SELECT tablename, rowsecurity
FROM   pg_tables
WHERE  tablename IN ('dc_decisions', 'dc_logs');

SELECT diagnostici — stato cron e notifiche
Query utili per verificare lo stato dei cron job e della coda notifiche dopo aver applicato il fix.
SQL pg_cron · pg_net 
-- Stato dei job cron
SELECT jobid, jobname, schedule, active, command
FROM   cron.job
ORDER  BY jobname;

-- Ultime esecuzioni (successo / errore)
SELECT runid, jobid, status, start_time, end_time, return_message
FROM   cron.job_run_details
ORDER  BY start_time DESC
LIMIT  20;

-- Coda notifiche (ultimi record)
SELECT *
FROM   notification_queue
ORDER  BY created_at DESC
LIMIT  20;

-- Richieste HTTP uscenti (pg_net)
SELECT id, method, url, status_code, error_msg, created
FROM   net._http_response
ORDER  BY created DESC
LIMIT  10;
